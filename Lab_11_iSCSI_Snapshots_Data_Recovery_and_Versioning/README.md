# Lab 11: iSCSI Volume Snapshots & Point-in-Time Recovery 🔄

## 📋 Описание
Лабораторная работа по управлению мгновенными снимками (snapshots) блочных томов в среде сетевых хранилищ. Практика охватывает создание снапшотов до и после модификации данных, восстановление тома из резервной точки, параллельное подключение снимков для анализа и полную очистку тестовой инфраструктуры.

## 🎯 Цели
1. Развернуть iSCSI-инфраструктуру (LUN, Host, Initiator) и подключить том к системе
2. Заполнить том тестовыми данными и зафиксировать исходное состояние
3. Создать мгновенный снимок (Snapshot 1) как точку восстановления
4. Внести изменения в данные на основном томе
5. Создать второй снимок (Snapshot 2) после модификации
6. Выполнить восстановление тома из Snapshot 1 (откат к исходному состоянию)
7. Перевести диск в offline/online режим в `diskmgmt.msc` для применения изменений
8. Проверить целостность данных после восстановления
9. Подключить Snapshot 2 как независимый том для сравнения версий
10. Выполнить полную очистку: удаление LUN, Host, всех снимков и инициатора

##  Выполненные задания

### Этап 1: Подготовка среды и создание LUN
- Создан LUN в системе хранения, настроены права доступа
- Зарегистрирован iSCSI Host и Initiator, выполнен маппинг
- Том подключён к Windows-клиенту, инициализирован, отформатирован, присвоена буква диска
- Заполнены тестовые файлы/папки для фиксации исходного состояния

### Этап 2: Создание Snapshot 1 (Point-in-Time A)
- Выполнено создание мгновенного снимка через интерфейс управления хранилищем
- Зафиксировано время создания и идентификатор снапшота
- Подтверждена доступность снимка в списке резервных точек

### Этап 3: Модификация данных и создание Snapshot 2
- На основном томе изменены/удалены/добавлены файлы (имитация рабочей нагрузки или ошибки)
- Создан Snapshot 2, фиксирующий состояние после изменений
- Проверена цепочка снимков: `Snapshot 1 → Active Volume → Snapshot 2`

### Этап 4: Восстановление из Snapshot 1
- Запущен процесс восстановления (Restore/Rollback) из Snapshot 1
- Диск переведён в `Offline` → `Online` через `diskmgmt.msc` для применения метаданных
- Проверена файловая система: структура и содержимое соответствуют исходному состоянию

### Этап 5: Параллельное подключение Snapshot 2
- Snapshot 2 подключён как отдельный read-only или clone-том
- Выполнено сравнение содержимого с основным и восстановленным томами
- Подтверждена возможность анализа исторических версий без влияния на production-данные

### Этап 6: Очистка инфраструктуры
- Отключены все тома, удалены снимки
- Удалены LUN, Host, iSCSI-инициатор
- Виртуальная машина завершена (`Пуск → Завершение работы`)

##  Ключевые выводы

| Технология | Назначение | Плюсы | Ограничения |
|------------|------------|-------|-------------|
| **Instant Snapshot** | Фиксация состояния тома в конкретный момент | Мгновенное создание, низкое потребление места (CoW/RoW), быстрый откат | Не заменяет полноценный бэкап (зависит от исходного тома), требует контроля цепочки |
| **Restore/Rollback** | Возврат к точке восстановления | Минимальный RTO, сохранение структуры ФС | Прерывает активные сессии, требует offline-перезапуска диска |
| **Snapshot Mount** | Анализ исторических версий | Безопасное сравнение, тестирование без риска | Требует дополнительных ресурсов хранилища |

### Почему это важно для сисадмина:
**Защита от human error** — мгновенный откат при ошибочном удалении/изменении файлов  
**Тестирование обновлений** — создание снапшота перед patch-днём, быстрый rollback при сбое  
**Соответствие SLA** — снижение RTO (Recovery Time Objective) до минут вместо часов  
**Enterprise-стандарт** — снапшоты используются в СХД (NetApp, Dell EMC, Huawei, YADRO) и гиперконвергентных платформах  

## 🔧 Использованные инструменты
- **Storage Management GUI/CLI** — управление LUN, снимками, маппингом
- **iSCSI Initiator** (`iscsiadm` / Windows iSCSI Initiator) — подключение блочных устройств
- **Disk Management** (`diskmgmt.msc`) — инициализация, online/offline, управление буквами
- **File Explorer / `dir` / `Get-ChildItem`** — верификация целостности данных
- **Snapshot Chain Viewer** — контроль зависимостей и точек восстановления

##  Теоретическая база

### Snapshots vs Backups
| Критерий | Snapshot | Backup |
|----------|----------|--------|
| **Скорость создания** | Мгновенно (секунды) | Минуты/часы |
| **Зависимость** | Привязан к исходному тому | Автономная копия |
| **Назначение** | Быстрый откат, тестирование, версионирование | Долгосрочное хранение, DR, архивация |
| **RPO/RTO** | Низкий RTO, средний RPO | Высокий RPO, высокий RTO |

### Механизмы работы снапшотов:
- **Copy-on-Write (CoW)** — при изменении блока оригинал копируется в область снапшота до записи новых данных. Надёжно, но может снижать производительность записи.
- **Redirect-on-Write (RoW)** — новые данные пишутся в свободное место, метаданные снапшота указывают на старые блоки. Выше производительность, сложнее управление фрагментацией.

### Best Practices в production:
1. Не хранить снапшоты дольше 7-14 дней без миграции в бэкап
2. Мониторить заполнение snapshot-пула (переполнение → крах тома)
3. Использовать снапшоты как первый слой защиты, а не замену offsite-бэкапам
4. Тестировать процедуру восстановления ежеквартально
